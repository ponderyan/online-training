# Phase A 设计决策记录（2026-06-20）

> 本文件记录 DTC 真实流程差距分析、Phase 分步决策依据、以及各设计选项的取舍理由。
> 用于后续问题回溯和 Phase B/C 设计参考。

## 1. DTC 真实流程 vs 系统现状（差距分析）

### 1.1 调研来源
桌面文件夹 `以DTC为例的培训（To 协会）/` 中的真实文件：
- 报名表（含招生机构列、补考备注）
- 考试成绩统计（分题型成绩明细）
- 证书申请信息（证书编号+有效期+按机构分发）
- 考场纪律文件（开卷考试规则）
- 试卷（单选40+判断10+填空10+简答10+案例20）
- 培训总结新闻 + 致学员信
- 某次全员满分/超高分的不正常考试成绩单（反映中百信系统数据问题）

### 1.2 逐环节对比

| 环节 | DTC 真实做法 | 系统现状（Phase 0） | 差距 |
|------|-------------|-------------------|------|
| 招生通知 | 发通知到机构 → 机构汇总报名 | ❌ 无 | 需要培训班+招生机构 |
| 学员报名 | 报名表含：姓名/手机/单位/职务/机构来源 | ⚠️ 学员管理有基础信息 | 缺少报名时间、机构来源、缴费状态 |
| 开班备案 | 培训班有编号、时间、负责人 | ❌ 无培训班概念 | 需要 TrainingProgram |
| 出题组卷 | 出题人命题→审核→组卷 | ✅ 题库+试卷 | 缺少命题人字段 |
| 考试 | 现场考试（开卷/闭卷+考场纪律） | ✅ Exam 模块可用 | 缺少开卷规则配置、迟到/交卷限制 |
| 阅卷 | 客观题自动+主观题人工评分 | ⚠️ Grading 模块有但缺盲批 | 缺少盲批、分题指派、scoreBreakdown |
| 成绩单 | 分题型统计（单选40/判断10/填空10/简答10/案例20） | ❌ 无成绩单页 | 需要 transcript 页+scoreBreakdown |
| 分数调整 | 人工调整+原始分保留 | ✅ ScoreAuditLog 已有 | 审计日志完整，可用 |
| 证书申请 | 机构统一申请→管理员发证 | ⚠️ Certificate 有但缺审批 | 需要 approvalStatus 审批流 |
| 证书分发 | 按机构分发→机构发学员 | ❌ 无 | 后期做 |
| 培训总结 | 写新闻稿→致学员信 | 明确不纳入系统（离线） | 不纳入 |
| 证书盖章邮寄 | 物理操作 | 明确不纳入系统（离线） | 不纳入 |

**核心结论**：当前系统只有一个"考试"环节可用，其余 8 个环节都缺失。

### 1.3 优先级判断依据
- **先做流程刚需**：培训班→报名→考试→阅卷→成绩单→证书审批
- **推迟的**：讲师管理（DTC案例中讲师是固定的，不做也能跑通流程）、排课（协会培训课表简单，Excel可管）、评价（非必须）
- **不做的**：课前调研（Word模板）、纸质签到（现场物理签到）、证书物理处理（打印/盖章/邮寄）、培训总结新闻（Word文档）

## 2. Phase 分步决策

### 2.1 为什么分 Phase A/B/C 而不是一步到位

**原则**：让系统先跑通最小可用闭环，再逐步增强。

**风险判断**：
- 一次性做完讲师+排课+评价，逻辑复杂且容易出bug
- 核心价值在"考试管理→证书发放"这条线，讲师管理不阻塞业务
- DTC 一期实际业务中，讲师是外部聘请的固定人员，不需要系统管

### 2.2 Phase A 范围（当前执行）

**核心目标**：用 TrainingProgram 把 Exam→Certificate 串起来

包含：
- ✅ 培训班管理（TrainingProgram CRUD + 状态机）
- ✅ 招生机构（EnrollmentAgency 基础CRUD）
- ✅ 学员报名（ProgramEnrollment + 缴费）
- ✅ 考试关联培训班（Exam.programId）
- ✅ 考试规则配置（开卷/迟到/交卷/通过线）
- ✅ 分题型成绩明细（scoreBreakdown JSON）
- ✅ 成绩单页面 + CSV导出
- ✅ 证书审批流程（DRAFT→APPROVED）
- ✅ 证书申请（学员端发起）

### 2.3 Phase B 范围（后续）

**核心目标**：阅卷增强 + 讲师基础管理

- 盲批模式（阅卷人看不见学员信息）
- 分题指派（题A给讲师李，题B给讲师王）
- 阅卷进度仪表盘
- 二审/复核机制
- 成绩申诉流程
- Instructor 模型（讲师基础信息+级别）
- Course 模型（课程大纲）

### 2.4 Phase C（更远）

**核心目标**：排课 + 评价 + 增强报表
- 排课表（时间+地点+讲师）
- 讲师评价（学员参评）
- 课程评价
- 培训数据统计
- 证书公开防伪查询

## 3. 关键设计决策与取舍

### 3.1 TrainingProgram 的定位

**决策**：培训班是业务主心骨，不是考试的附属。

考试可以独立存在（不绑定培训班），也可以属于一个培训班（通过 `programId?`）。
培训班可以有多个考试（结业考+补考）。

**理由**：协会场景下，培训班是行政管理单元，考试只是流程中的一环。
如果考试无法独立存在（比如 ITSS 单独组织的认证考试），系统就不够灵活。

### 3.2 状态机简化

**决策**：Phase A 使用精简状态机。

| 实体 | Phase A 状态 | 完整版（未来） |
|------|-------------|--------------|
| TrainingProgram | DRAFT→ENROLLING→IN_PROGRESS→FINISHED→CANCELLED | 需加 PENDING_REVIEW/APPROVED/CLOSED |
| Exam | DRAFT→PUBLISHED→IN_PROGRESS→FINISHED→CANCELLED | 需加 GRADING/PUBLISHED_GRADES 等 |
| ExamSession | PENDING→GRADING→GRADED→PUBLISHED→ADJUSTED | 需加 CONFIRMED/APPEALED |

**理由**：协会备案审批流程和成绩锁定机制在 Phase A 不急，先跑通核心流程。

### 3.3 scoreBreakdown 用 JSON 而非关联表

**决策**：JSON 字段，不建分表。

```json
{
  "SINGLE_CHOICE": { "total": 40, "earned": 32, "count": 20, "correct": 16 },
  "TRUE_FALSE": { "total": 10, "earned": 8, "count": 10, "correct": 8 },
  "SHORT_ANSWER": { "total": 20, "earned": 15, "count": 2, "correct": 0 }
}
```

**理由**：
- 分题型统计是"查"多过"写"，JSON 查询简单
- DTC 案例中题型 + 满分是固定的（单40判10填10简10案20）
- 不需要单独对每个题型做审计/调整（调整的是总分不是题型分）
- 避免 N+1 查询，减少复杂度

### 3.4 证书审批流程设计

**决策**：三步走——草稿(DRAFT)→通过(APPROVED)→撤销(REVOKED)

**业务流程对应**：
1. 成绩发布 → 自动创建证书草稿（不自动发证）
2. 管理员审核草稿 → 审批通过（正式发证，生成编号）
3. 可撤销（撤回证书）

**理由**：
- 协会真实流程中，成绩出来到发证之间有审核环节
- 不是所有通过考试的人都一定给证书（如欠费、未结业）
- 草稿让管理员先看再决定

### 3.5 证书自动创建时机

**决策**：只在「关联了培训班的考试」发布成绩时自动创建证书草稿。

独立考试（无 programId）不自动创建，需要手动或通过证书申请发起。

**理由**：
- 培训班是发证业务的触发条件
- 独立考试可能只是测试/模拟，不应自动发证

### 3.6 报名模式

**决策**：Phase A 只做「管理员后台添加学员」模式。

**不做**：学员自助注册报名（需验证码、身份认证等）。

**理由**：
- DTC 真实流程是机构汇总后统一报名
- 学员自助报名的认证流程复杂
- Phase A 先保障管理员能干活

### 3.7 考试成绩单

**决策**：新页面 `/exams/[id]/transcript`，含分题型明细 + CSV 导出。

**理由**：
- DTC 真实场景中，每次考完要出成绩单给协会存档
- 包含按机构统计的需求
- CSV 导出是为了发给机构负责人
- 打印功能用于线下会议审核

## 5. 重要业务规则

### 5.1 TrainingProgram 自动编号
- 格式：`DT-TC-{YYYY}-{NNN}`
- 每年从 001 重新开始
- 示例：DT-TC-2026-001, DT-TC-2026-002

### 5.2 教考分离原则（Phase B 实现）
- 命题人不可阅卷
- 阅卷人不可见学员信息（盲批）
- 考务不可见试题内容

### 5.3 数据不可篡改
- 每次成绩调整记录 ScoreAuditLog
- 证书审批记录 CertificateApprovalLog
- 所有流转有追溯

### 5.4 用户角色
- SUPER_ADMIN: 超管
- ORG_ADMIN: 机构管理员（协会管理员）
- LECTURER: 讲师（可出题可阅卷）
- PROCTOR: 监考员
- STUDENT: 学员
