那些我犯过的错（以及它们教会我的事）

2026年7月1日

 

某个深夜，我盯着一个报错日志发呆。同一个接口，昨天还好好的，今天返回了完全不同的结果。排查了两轮才发现——代码是对的，但编译产物没更新。我修了源码，忘了重建。

 

这不是第一次了。

 

错误 1：17 个编译错误

 

有一次我改了一个数据库查询，加了一个字段，但没在数据模型里声明它。TypeScript 在编译时用 17 个红色错误告诉了我这一点——不是在运行时报错，是在编译时。那是我第一次理解什么叫"类型系统的价值"。

 

在传统的动态语言开发中，这种错误会在运行时出现，可能被用户发现才意识到出了问题。静态类型检查把这些错误提前到了开发阶段。代价是写代码的时候要多想一步，但换来的是运行时少一次崩溃。

 

从那以后，我改任何代码之前会先确认三件事：数据模型改了吗？接口类型对了吗？编译过了吗？

 

错误 2：一个函数调用了不存在的方法

 

前端报了一个运行时错误："xxx.find is not a function"。查了两层才发现原因：后端 API 返回的是一个对象 { items: [], total: 20 }，但前端代码把它当数组直接用了，以为它就是列表本身。

 

这个问题本质上是我在前端代码里做了"假设"——假设 API 返回的东西跟我想的一样，没去验证实际结构。

 

现在我对 API 的返回结构会多一层检查，不会盲目相信"应该就是这个样子"。如果数据是对象，就用 .items 取列表；如果是数组，就直接用。不确定的时候，加个 Array.isArray() 判断。

 

错误 3：写了新功能，忘了老功能

 

有一次我替换了一个功能模块的实现方式，把旧的代码路径废弃了。但我忘了确认旧模块在其他地方还有没有被引用。结果是一个不相关的页面调了旧接口，返回了空数据。

 

这让我学会了一件事：改代码之前，先 grep 一遍所有引用。你觉得"这肯定没人用了"，往往是错的。

 

为什么我要写下这些

 

不是因为读者需要知道我犯过什么错。而是因为如果你在做一个系统，你也会犯这些错。而且你会觉得"只有我在犯"。不是的，大家都在犯。

 

错本身不是问题。问题是你有没有从错里学到东西，并且在学到之后不再犯同样的错。我做到了第三点，才敢把这些写下来。
